# PROGETTO: AI Suite per Amministratori di Condominio — Italia

## CONTESTO STRATEGICO

Stai costruendo una startup B2B SaaS per il mercato italiano degli amministratori di condominio (35.000 studi, €300M+ TAM potenziale).

Il problema core: l'amministratore medio perde 5+ ore/giorno rispondendo alle stesse domande dei condòmini. Zero innovazione nel settore — i software attuali (Danea, Sistemi, Buffetti) sono gestionali anni '90, non assistenti intelligenti.

La soluzione: piattaforma AI che gestisce comunicazione condòmini (email + WhatsApp), fornisce knowledge base interna per operatori studio, e automatizza risposte su normativa + regolamenti condominiali.

Tu sei un technical founder con mentalità 0.001% elite. Ogni decisione deve essere: validabile, scalabile, difendibile.

## FASE 1: BRAND IDENTITY & NAMING

### Obiettivo
Creare 5 proposte di naming + logo che funzionino per il mercato italiano degli amministratori di condominio. Il target sono professionisti 40-60 anni, mediamente conservatori, ma aperti a soluzioni che fanno risparmiare tempo.

### Task 1.1: Analisi Naming

Crea file `/01-strategy/brand/names-analysis.md` con:

Per ogni nome (5 totali), analizza:

1. **Nome proposto** (possono essere italiani o inglesi, purché pronunciabili da italiano medio)
2. **Razionale strategico** — perché funziona per questo target
3. **Disponibilità dominio** — verifica .it e .com (indica come verificare)
4. **Connotazioni** — cosa evoca, rischi di fraintendimento
5. **Scalabilità** — funziona se espandiamo ad altri verticali?
6. **Memorabilità** — test: lo ricordi dopo 5 minuti?

Criteri per i nomi:
- Devono comunicare: affidabilità, modernità, semplicità
- Evitare: troppo tech/startup-y (spaventa il target), giochi di parole forzati
- Bonus: se contiene riferimento a "condominio" o "amministrazione" senza essere banale

## FASE 4: PRODUCT SPECIFICATION

### Task 4.1: Product Requirements Document (PRD)

Crea `/02-product/specs/PRD.md` con:

#### Overview Prodotto

**Vision:** [1 frase che descrive il mondo dopo che esistiamo]

**Mission:** [come ci arriviamo]

**Principi Guida:**
1. Semplicità > Feature
2. Integrazione > Sostituzione
3. Valore immediato > Promesse future

#### Utenti e Ruoli

| Ruolo | Descrizione | Permessi |
|-------|-------------|----------|
| Admin Studio | Titolare/responsabile | Full access |
| Operatore | Segretaria/collaboratore | Gestione comunicazioni, no billing |
| Condòmino | Utente finale | Solo proprio condominio, chat, documenti |

#### Feature Set MVP (P0)

**F1: Dashboard Principale**
- Obiettivo: Vista unificata attività pending
- User story: "Come operatore, voglio vedere tutti i messaggi non gestiti in un posto"
- Requisiti funzionali:
  - Lista messaggi WhatsApp in arrivo
  - Lista email in arrivo
  - Filtri per condominio, urgenza, stato
  - Azioni rapide (rispondi, assegna, chiudi)
- Requisiti non funzionali:
  - Caricamento < 2 sec
  - Mobile responsive

**F2: WhatsApp Integration**
- Obiettivo: Gestire WhatsApp Business da dashboard
- User story: "Come operatore, voglio rispondere a WhatsApp senza usare il telefono"
- Requisiti funzionali:
  - Ricezione messaggi in real-time
  - Invio messaggi (testo, immagini, documenti)
  - Template messaggi pre-approvati
  - Auto-reply fuori orario
  - Suggerimento risposta AI
- Requisiti non funzionali:
  - Latenza < 5 sec
  - Compliance WhatsApp Business API ToS
- Dipendenze: WhatsApp Business API account

**F3: Email Integration**
- Obiettivo: Centralizzare email condomìni
- User story: "Come operatore, voglio vedere email e WhatsApp insieme"
- Requisiti funzionali:
  - Connessione IMAP/SMTP o Gmail/Outlook API
  - Threading conversazioni
  - Suggerimento risposta AI
  - Template email
- Requisiti non funzionali:
  - Sync ogni 2 minuti max
  - Supporto allegati fino 25MB

**F4: AI Response Engine**
- Obiettivo: Suggerire/generare risposte accurate
- User story: "Come operatore, voglio che l'AI scriva la risposta e io approvo"
- Requisiti funzionali:
  - Analisi messaggio in arrivo
  - Generazione risposta contestuale
  - Fonti: normativa, regolamento condominio, storico comunicazioni
  - Confidence score
  - Edit before send
  - Feedback loop (risposta usata/modificata/rifiutata)
- Requisiti non funzionali:
  - Response generation < 10 sec
  - Accuracy > 85% (misurabile via feedback)

**F5: Knowledge Base Interna**
- Obiettivo: Chat per operatori su normativa e regolamenti
- User story: "Come segretaria, voglio chiedere all'AI se posso rispondere X a un condòmino"
- Requisiti funzionali:
  - Chat interface interno (non visibile a condòmini)
  - RAG su: Codice Civile artt. condominio, normativa regionale, regolamento specifico condominio caricato
  - Citazione fonti nelle risposte
  - Storico conversazioni
- Requisiti non funzionali:
  - Response < 15 sec
  - Fonti sempre linkate

**F6: Document Management Base**
- Obiettivo: Caricare regolamenti per RAG
- User story: "Come admin, voglio caricare il regolamento del condominio X così l'AI lo conosce"
- Requisiti funzionali:
  - Upload PDF/Word
  - OCR se necessario
  - Associazione documento ↔ condominio
  - Versioning base
- Requisiti non funzionali:
  - Upload max 50MB
  - Processing < 5 min

#### Feature Set Post-MVP (P1)

- Automation rules (if X then Y)
- Scheduling messaggi
- Report analytics comunicazioni
- Integrazione gestionale (read-only)
- Multi-lingua (per condomìni con stranieri)
- Mobile app nativa

#### Feature Set Future (P2)

- Gestione assemblee (convocazioni, deleghe, verbali)
- Portale condòmino self-service
- Integrazione pagamenti
- Chiamate vocali con AI

#### Requisiti Non Funzionali Globali

| Categoria | Requisito |
|-----------|-----------|
| Performance | Page load < 3s, API response < 500ms |
| Availability | 99.5% uptime |
| Security | GDPR compliant, encryption at rest e in transit, 2FA |
| Scalability | Support 100 studi concurrent in MVP |
| Backup | Daily backup, 30 days retention |

**OUTPUT ATTESO:** PRD completo e dettagliato.
